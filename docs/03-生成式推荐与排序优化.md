# 生成式推荐与排序优化

## 1. 模型架构

### 1.1 基座模型

- **模型**：`Qwen/Qwen2.5-3B`（30 亿参数）
- **加载方式**：`AutoModelForCausalLM.from_pretrained`，`torch.float16` 半精度
- **设备映射**：可配置（默认 `cuda`）

### 1.2 LoRA 微调

- **适配器加载**：`PeftModel.from_pretrained(base, model_lora_path)`
- **权重路径**：`./models/checkpoints/qwen25_rec_lora`
- **训练方法**：SFT（监督微调）+ GRPO（组相对策略优化）
- **模型版本标识**：`qwen2.5-3b-sft-grpo-v1`

### 1.3 训练方法说明

**SFT 阶段**：
- 训练数据：`(历史 SID 序列, 下一个 SID)` 配对
- 使用 LoRA 适配器微调，冻结基座参数
- 目标：让模型学会从行为序列预测下一个商品的 SID

**GRPO 阶段**：
- 基于强化学习的优化，使用组内相对奖励
- 约束解码：惩罚生成不在合法词汇表中的 SID
- 排序感知奖励：奖励正确排序相关商品
- 目标：提升推荐质量，减少无效输出

## 2. 推理流程

### 2.1 Prompt 构建

文件：`backend/models/recommender_inference.py`

```
[INST] Based on the user's browsing history (item semantic IDs):
History: 12_5_3_7 8_2_14_6 3_11_0_9
Predict the next items. Output semantic IDs only, one per line.
[/INST]
```

- 历史 SID 用空格分隔
- 模型输出每行一个 SID

### 2.2 Beam Search 参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `num_beams` | 20 | 束搜索宽度 |
| `num_return_sequences` | 20 | 返回所有 20 个候选序列 |
| `max_new_tokens` | 32 | 每个 beam 最大生成 token 数 |
| `early_stopping` | True | 生成 EOS 即停止 |
| `pad_token_id` | `eos_token_id` | 用 EOS 作填充 token |

### 2.3 候选生成逻辑

```python
outputs = model.generate(
    **inputs,
    num_beams=20,
    num_return_sequences=20,
    max_new_tokens=32,
    early_stopping=True,
)
```

后处理：
1. 遍历 20 个输出序列
2. 截掉 prompt 部分（用 `input_ids.shape[1]` 定位）
3. 解码生成的 token，取第一行文本作为 SID
4. 去重，收集到 `top_k=10` 个唯一 SID 即停止
5. 保持 beam search 评分顺序（beam 0 概率最高）

### 2.4 降级模式

PyTorch 不可用或模型加载失败时，进入模拟模式：
- 用 `hash(tuple(history_sids))` 作为随机种子
- 生成确定性伪随机 SID（格式 `{0-15}_{0-15}_{0-15}_{0-15}`）
- 系统可完整运行，但推荐结果无实际意义

## 3. 推荐流水线

### 3.1 完整流程

文件：`backend/services/recommend_pipeline.py` + `backend/api/routers/recommend.py`

```
用户历史 ASIN 列表
       ↓
Step 1: ASIN → SID（SIDService，三级查询）
       ↓
Step 2: SID 序列 → LLM 推理 → 候选 SID（Beam Search）
       ↓
Step 3: 候选 SID → ASIN（SIDService，反向查询）
       ↓
Step 4: ASIN → 商品详情（ESClient，三级回查）
       ↓
Step 5: 按模型评分排序，返回 Top-K
```

### 3.2 排序逻辑

排序完全由模型驱动：

```python
order = {asin: i for i, asin in enumerate(candidate_asins)}
products.sort(key=lambda p: order.get(p["asin"], 999))
```

- 模型输出的候选顺序即为最终排序
- beam search 的对数概率决定排名
- 无二次重排（不按价格、评分、人气等后处理）

### 3.3 缓存机制

| 缓存 | Key | TTL | 命中行为 |
|------|-----|-----|---------|
| 推荐结果 | `recommend:{user_id}:{md5(asins)}` | 5 分钟 | 直接返回缓存结果 |
| SID 映射 | `sid:{asin}` | 24 小时 | 跳过 Milvus/JSON 查询 |
| 商品详情 | `product:{asin}` | 1 小时 | 跳过 Milvus/ES 查询 |

推荐缓存 key 计算：对 history_asins 排序后取 MD5（截断 12 字符），确保相同商品集合（不同顺序）命中同一缓存。

## 4. 指标收集

### 4.1 运行时指标

文件：`backend/api/middleware/logging_middleware.py`

| 指标 | 类型 | 说明 |
|------|------|------|
| `recommend_request_total` | counter | 推荐请求总数 |
| `recommend_empty_count` | counter | 模型未生成有效候选的次数 |
| `sid_hit_rate` | ratio | SID 查询命中率 |
| `avg_recommend_latency_ms` | gauge | 推荐接口平均延迟 |
| `avg_model_inference_ms` | gauge | 模型推理平均耗时 |
| `avg_es_query_ms` | gauge | 商品回查平均耗时 |

通过 `GET /metrics` 端点暴露。

### 4.2 离线评估指标

- **HR@10**（Hit Rate at 10）：约 0.42 — Top-10 推荐中命中用户实际交互的比例
- **NDCG@10**（Normalized Discounted Cumulative Gain at 10）：约 0.28 — 考虑排序位置的命中质量

## 5. 关键设计决策

1. **生成式而非检索式**：传统推荐系统（协同过滤、双塔模型）从候选池中打分排序，本系统让 LLM 直接"生成"商品 ID，将推荐转化为序列生成问题
2. **SID 作为离散 token**：将连续 embedding 空间量化为离散 token，使 LLM 的自回归生成能力能直接应用于推荐
3. **Beam Search 保留多样性**：20 束搜索 + 去重，既保证候选质量又避免重复推荐
4. **纯模型排序**：不引入业务规则重排，完全信任模型的评分排序能力
5. **三级容错**：SID 查询和商品回查均有 Redis → Milvus/ES → JSON/占位符 的降级路径

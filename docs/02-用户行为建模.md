# 用户行为建模

## 1. 离线行为序列构建

### 1.1 数据来源

原始数据：Amazon Reviews 的 `Industrial_and_Scientific.jsonl`，每行包含 `reviewerID`、`asin`、`unixReviewTime`。

### 1.2 处理流程

脚本：`backend/data/scripts/preprocess_interactions.py`

1. **按用户聚合**：用 `defaultdict(list)` 按 `reviewerID` 分组，每条记录保留 `asin` 和 `unixReviewTime`
2. **时间排序**：每组按 `unixReviewTime` 升序排列，保证行为序列的时间顺序
3. **冷启动过滤**：丢弃交互次数 < `min_seq_len=5` 的用户（信号不足）
4. **序列截断**：只保留最近 `max_seq_len=50` 条行为（控制模型输入长度）

**输出**：`data/processed/user_interactions.jsonl`

```json
{"user_id": "A1B2C3D4E5", "sequence": ["ASIN1", "ASIN2", ..., "ASIN50"]}
```

### 1.3 设计意图

此文件是**模型训练样本**，不是运行时数据。每条记录代表一个用户的完整交互历史，用于训练模型学习"给定历史 ASIN 序列 → 预测下一个 ASIN"的模式。

## 2. 语义 ID（SID）系统

### 2.1 什么是 SID

SID（Semantic ID）是商品的离散语义标识符，由 RQ-VAE 模型生成。格式为 4 个下划线分隔的整数，如 `"12_5_3_7"`，每个整数对应一层残差量化器的码本索引（范围 0-15）。

SID 的核心作用：将连续的文本 embedding 空间映射为离散的 token 空间，使 LLM 能像预测下一个词一样预测下一个商品。

### 2.2 RQ-VAE 编码器

文件：`backend/models/rqvae_encoder.py`

**架构**：
- 输入：商品文本（标题 + 描述前 200 字符）
- 通过 Sentence-Transformer 编码为 384 维向量
- 经过 4 层残差量化（Residual Quantization）：
  - 每层有一个码本（codebook），包含 16 个学习到的原型向量
  - 第一层量化原始向量，后续层量化前一层的残差
  - 每层输出一个码本索引（0-15）
- 输出：4 个索引拼接为 SID 字符串 `"a_b_c_d"`

**理论词汇量**：16^4 = 65,536 个可能的 SID。实际数据集使用了 60,203 个唯一 SID。

**碰撞处理**：当两个商品映射到同一 SID 时，`sid2asin` 字典取第一个（`setdefault`）。

### 2.3 SID 映射生成

脚本：`backend/data/scripts/build_sid_mapping.py`

1. 加载 RQ-VAE 编码器
2. 读取 `products.jsonl` 中所有商品
3. 拼接文本 `"{title} {description[:200]}"` 作为编码输入
4. 批量编码（256 条/批），生成 SID
5. 构建双向映射：`asin2sid` 和 `sid2asin`
6. 输出 `sid_mapping.json`

### 2.4 SID 查询服务

服务：`backend/services/sid_service.py`

**三级查询策略**：

| 层级 | 存储 | TTL | 说明 |
|------|------|-----|------|
| L1 | Redis `sid:{asin}` | 24h | 热点缓存 |
| L2 | Milvus `sid_mapping` 集合 | — | 主存储 |
| L3 | `sid_mapping.json` 文件 | — | 兜底方案 |

**核心方法**：
- `asins_to_sids(asins)` — 正向查询（ASIN → SID），结果回填 Redis
- `sids_to_asins(sids)` — 反向查询（SID → ASIN），跳过 Redis 缓存
- `is_valid_sid(sid)` — 同步校验 SID 是否有效（用于模型推理过滤）
- `get_all_valid_sids()` — 返回所有合法 SID（用于模型加载）

## 3. 实时行为采集（前端）

### 3.1 行为存储

文件：`frontend/src/store/behaviorStore.ts`

使用 Zustand + `persist` 中间件，行为记录持久化到 `localStorage`（key: `"behavior-history"`）。

**状态结构**：

```typescript
interface BehaviorRecord {
  asin: string;       // 商品 ID
  title: string;      // 商品标题
  action: "view" | "click" | "purchase";  // 行为类型
  timestamp: number;  // Unix 时间戳
}
```

**核心逻辑**：
- `addRecord(record)` — 按 ASIN 去重（最新交互覆盖旧记录），插入头部，截断到 50 条
- `removeRecord(asin)` — 删除指定商品
- `clearAll()` — 清空历史
- `historyAsins()` — 返回 ASIN 列表，按时间升序排列（最旧在前）

### 3.2 行为触发

文件：`frontend/src/components/recommendation/RecommendFeed.tsx`

| 用户操作 | 记录行为 | 后端上报 |
|---------|---------|---------|
| 点击推荐商品 | `click` | `POST /behavior` |
| 点击"加入浏览历史" | `view` | `POST /behavior` |

### 3.3 响应式推荐触发

文件：`frontend/src/hooks/useRecommend.ts`

```typescript
const { history } = useBehaviorStore();
useQuery({
  queryKey: ["recommend", userId, history],  // history 变化 → 重新请求
  queryFn: () => fetchRecommendations({ user_id: userId, history_asins: history }),
  enabled: history.length > 0,
});
```

**关键设计**：history 数组作为 TanStack Query 的 queryKey，任何行为变更自动触发推荐刷新，形成实时个性化反馈循环。

### 3.4 行为上报 API

接口：`POST /api/v1/behavior`

```python
class BehaviorRequest(BaseModel):
    user_id: str
    asin: str
    action_type: str  # view, click, purchase
```

当前为日志记录桩，结构化日志通过 structlog 输出，预留未来接入行为持久化管道。

## 4. 端到端行为建模流程

```
离线阶段：
  原始评论 → 用户行为序列 → 训练样本
                              ↓
                     RQ-VAE 编码 → SID 映射

在线阶段：
  用户浏览/点击 → behaviorStore (localStorage)
                        ↓
                  historyAsins() → POST /recommend
                        ↓
              ASIN → SID → LLM 推理 → SID → ASIN → 商品详情
```

### 关键架构决策

1. **客户端行为存储**：历史记录存在浏览器 localStorage，不依赖服务端数据库，简化架构但设备隔离
2. **SID 作为模型语言**：LLM 从不接触原始 ASIN 或文本，完全在 SID 空间操作，实现跨商品泛化
3. **三级 SID 查询**：Redis → Milvus → JSON，兼顾性能与容错
4. **history 作为 queryKey**：行为变更自动触发推荐刷新，无需手动轮询
